# ТЗ: Мобильные приложения для Android и iOS (VPN-клиент)

## 1. Цель

Создать кросс‑платформенные клиентские приложения (Android и iOS) как оболочку для VPN‑сервиса:
- авторизация в ЛК;
- регистрация устройства (`device_id`);
- получение сервера и JWT через `POST /vpn/token`;
- подключение к TrustTunnel;
- сбор/отправка метрик;
- показ статуса и уведомлений.

Ручная конфигурация (TOML, текстовые конфиги) **не используется**. Конфигурацию формирует ЛК, клиент получает её автоматически.

---

## 2. Архитектура и стек

### 2.1 Рекомендуемый стек
- **React Native (TypeScript)** — единый UI и бизнес‑логика.
- **Native bridge / модуль VPN**:
  - iOS: Network Extension + Keychain;
  - Android: VpnService + Keystore/EncryptedSharedPreferences.
- **HTTP API слой**: REST клиент с поддержкой refresh токенов.
- **Push**: APNS (iOS) и FCM (Android).
- **Аналитика/метрики**: внутренний endpoint ЛК `/api/v1/metrics/client`.

### 2.2 Логические модули приложения
1. `AuthModule` — вход, refresh, logout, multi‑account.
2. `DeviceModule` — генерация/хранение/регистрация `device_id`.
3. `VpnSessionModule` — получение `vpn_jwt`, запуск/остановка туннеля, переподключение.
4. `MetricsModule` — сбор RED + ресурсы + действия пользователя.
5. `PushModule` — регистрация push token, хранение, inbox.
6. `UpdateModule` — проверка версии, forced update.
7. `UI` — экраны авторизации, VPN, статуса, уведомлений, аккаунтов.

## 3. Функциональные требования

### 3.1 Авторизация и управление устройствами

#### Авторизация
- Поддерживаются методы ЛК: e‑mail/пароль и OAuth.
- После логина клиент получает:
  - `access_token`;
  - `refresh_token`.
- Токены сохраняются только в безопасном хранилище.

#### Регистрация устройства
- При первом запуске генерируется UUID `device_id`.
- `device_id` регистрируется в ЛК и привязывается к аккаунту.
- ЛК проверяет подписку/лимиты и возвращает `device_user`.

#### Мультиаккаунт
- Хранение нескольких аккаунтов на устройстве.
- Активным может быть только один аккаунт в рамках одной VPN-сессии.
- При logout удаляются локальные ключи и токены выбранного аккаунта.

### 3.2 Получение конфигурации и подключение VPN

Перед каждым подключением клиент вызывает `POST /vpn/token` с `device_id`.

Ожидаемый ответ:
- `endpoint.address` (`IP:port`);
- `endpoint.hostname` (SNI);
- `endpoint.protocol` (например, `http2`);
- `endpoint.dns`;
- `vpn_username`;
- `vpn_jwt`;
- `expires_at`;
- дополнительные TLS‑параметры (опционально).

Подключение:
- TCP/TLS/HTTP2 к `endpoint.address`;
- SNI = `endpoint.hostname`;
- Basic Auth:
  - username = `vpn_username`;
  - password = `vpn_jwt`.

Обновление JWT:
- Автообновление до истечения `expires_at` (например, за 60–120 сек).
- При разрыве/ошибке — повторный `POST /vpn/token` и переподключение.

### 3.3 Сбор и отправка метрик

#### RED‑метрики VPN сессии
- время установления соединения;
- RTT/latency;
- download/upload throughput;
- количество ошибок и причины.

#### Ресурсные метрики
- CPU;
- память;
- диск;
- энергопотребление (где возможно через платформенные API).

#### Пользовательские действия (обезличенные)
- нажатие «Подключиться/Отключиться»;
- успешные/неуспешные сессии;
- ошибки авторизации/обновления токена.

#### Передача
- Endpoint ЛК: `POST /api/v1/metrics/client`.
- Отправка батчами (например, по окончании сессии или каждые N минут).
- Retry c backoff и дедупликацией event_id.

### 3.4 Push‑уведомления

- Запрос разрешения при первом запуске.
- Получение device token:
  - APNS для iOS;
  - FCM для Android.
- Передача token в ЛК вместе с `device_id` и текущим аккаунтом.
- Поддержка:
  - системного отображения (banner/sound);
  - внутреннего раздела «Уведомления»;
  - deep link действий (например, оплата).

### 3.5 UI/UX

Обязательные экраны:
1. **Авторизация** (вход, регистрация, восстановление пароля).
2. **Главный экран VPN**:
   - большая кнопка «Подключиться/Отключиться»;
   - текущий статус;
   - состояние подписки.
3. **Экран статуса**:
   - текущий сервер;
   - TTL JWT;
   - трафик;
   - базовые метрики.
4. **Уведомления** (in-app inbox).
5. **Аккаунты** (переключение/выход).

Принцип: минималистичный интерфейс без сложных ручных настроек VPN.

## 4. Нефункциональные требования

### 4.1 Безопасность
- Хранение секретов только в защищённом хранилище ОС.
- TLS pinning (рекомендуется).
- JWT не логировать в plaintext.
- Очистка чувствительных данных при logout/компрометации.
- Следование OWASP Mobile Top 10.

### 4.2 Надёжность
- Автопереподключение с ограниченным exponential backoff.
- Обработка оффлайн‑режима.
- Защита от гонок при refresh/access token.

### 4.3 Производительность
- Холодный старт: целевой SLA (например, <2–3 с до интерактивности).
- Минимизация нагрузки при активной VPN-сессии.

### 4.4 Обновления
- Проверка минимально поддерживаемой версии приложения.
- Forced update для критичных релизов.

## 5. API-контракты (предлагаемый минимум)

### 5.1 POST /vpn/token

Request:
```json
{
  "device_id": "uuid",
  "platform": "ios|android",
  "app_version": "1.0.0"
}
```

Response:
```json
{
  "endpoint": {
    "address": "203.0.113.10:443",
    "hostname": "edge-12.vpn.example.com",
    "protocol": "http2",
    "dns": "1.1.1.1"
  },
  "vpn_username": "dev_user_abc",
  "vpn_jwt": "eyJ...",
  "expires_at": "2026-01-10T10:15:00Z",
  "tls": {
    "alpn": ["h2"]
  }
}
```

Ошибки:
- `401` — невалидный access token;
- `402/403` — подписка неактивна/лимит устройств;
- `429` — rate limit;
- `5xx` — ошибка инфраструктуры.

### 5.2 POST /api/v1/metrics/client

Request (batch):
```json
{
  "device_id": "uuid",
  "session_id": "uuid",
  "events": [
    {
      "event_id": "uuid",
      "type": "vpn_connect",
      "ts": "2026-01-10T10:00:00Z",
      "payload": {
        "connect_ms": 820
      }
    }
  ]
}
```

Response:
```json
{
  "accepted": 1,
  "rejected": 0
}
```

## 6. Состояния VPN-сессии (state machine)

`idle -> preparing -> connecting -> connected -> reconnecting -> disconnected -> idle`

Дополнительно:
- `error_auth`
- `error_subscription`
- `error_network`

UI должен отображать состояние и рекомендуемое действие пользователя.

## 10. Критерии приёмки

1. Пользователь может войти (email/password или OAuth), подключить VPN одной кнопкой и увидеть статус.
2. `device_id` регистрируется и используется при каждом `POST /vpn/token`.
3. JWT обновляется автоматически, переподключение работает без ручного вмешательства.
4. Метрики отправляются на `/api/v1/metrics/client` и валидно принимаются сервером.
5. Push‑уведомления приходят и отображаются в системе и in‑app inbox.
6. Реализованы мультиаккаунт, logout с очисткой секретов и forced update.
